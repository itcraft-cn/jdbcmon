## 一、项目概述

JDBCMon 是一个高性能、可扩展的轻量级 JDBC 监控代理框架，采用动态代理模式实现零侵入的数据库操作监控。

### 模块结构

```
jdbcmon/
├── jdbcmon-core/           # 核心模块 - JDBC 代理包装、指标收集、监听器系统
├── jdbcmon-spring/         # Spring Boot 集成 - 自动配置、Actuator 端点
└── jdbcmon-test/           # 测试模块 - 单元测试、JMH 基准测试
```

## 二、架构与设计模式分析

### 2.1 设计模式使用

| 模式 | 应用位置 | 评价 |
|------|----------|------|
| **装饰器模式** | `MonitoredConnection`/`MonitoredStatement`/`MonitoredResultSet` | ✅ 优秀：透明包装 JDBC 对象，职责清晰 |
| **策略模式** | `ResultSetMonitor` 接口 + 三个实现类 | ✅ 优秀：HugeResultSetAction 决定不同监控策略 |
| **观察者模式** | `SqlExecutionListener` + `CompositeSqlListener` | ✅ 优秀：事件驱动，易于扩展 |
| **工厂模式** | `WrappedFactory` | ✅ 简洁：集中创建代理对象 |
| **建造者模式** | `WrappedConfig.Builder` | ✅ 标准：配置构建清晰 |
| **单例模式** | `BasicMetricsRecorder.INSTANCE` 等 | ✅ 合理：无状态对象复用 |

### 2.2 架构亮点

1. **分层清晰的监控体系**
   - `BasicMetricsRecorder` → `ExtendedMetricsRecorder` → `FullMetricsRecorder`
   - 三级指标收集策略，按需选择精度

2. **高性能设计**
   - `LongAdder` 用于高并发计数（优于 `AtomicLong`）
   - `ConcurrentHashMap` 用于线程安全的 SQL 指标存储
   - `CopyOnWriteArrayList` 用于监听器列表（读多写少场景）
   - 快速路径 API（`recordQueryFast`）避免重复查找

3. **SqlExecutionContext 对象池**
   - `ThreadLocal` 复用上下文对象，减少 GC 压力
   - 使用前 `reset()` 清零状态，防止数据污染

### 3.2 ⚠️ 需要改进的部分

#### 3.2.1 【高优先级】MonitoredResultSet 过于臃肿

**问题**: `MonitoredResultSet.java` 有 **1026 行**代码，违反单一职责原则

**影响**:
- 难以维护和测试
- 方法过多，扇出过大
- 不符合规范（文件不超过 2000 行，但已接近边界）

**建议**: 拆分为多个职责明确的类：
```
MonitoredResultSet (主类，~100 行)
├── ResultSetGetters (数据获取，~300 行)
├── ResultSetUpdaters (数据更新，~400 行)
├── ResultSetNavigation (导航方法，~100 行)
└── ResultSetMetaData (元数据方法，~50 行)
```

#### 3.2.2 【高优先级】MonitoredPreparedStatement 和 MonitoredCallableStatement 方法重复

**问题**: 
- `MonitoredPreparedStatement.java` 724 行
- `MonitoredCallableStatement.java` 683 行
- 两者大量参数设置方法完全重复（`setInt`, `setString` 等 50+ 方法）

**建议**: 抽取公共抽象基类 `BaseMonitoredStatement`:
```java
abstract class BaseMonitoredStatement {
    // 公共的 60+ 个参数设置方法
    void setInt(int index, int x) { delegate.setInt(index, x); }
    void setString(int index, String x) { delegate.setString(index, x); }
    // ...
}

class MonitoredPreparedStatement extends BaseMonitoredStatement implements PreparedStatement {
    // 仅需实现 PreparedStatement 特有方法
}
```

#### 3.2.3 【中优先级】MonitoredConnection 事务监控不完整

**问题**: `MonitoredConnection.java:149-166`
```java
@Override
public void close() throws java.sql.SQLException {
    delegate.close();  // ❌ 缺少连接关闭监控
}

@Override
public void setAutoCommit(boolean autoCommit) throws java.sql.SQLException {
    delegate.setAutoCommit(autoCommit);  // ❌ 缺少事务模式变更监控
}
```

**影响**: 无法追踪连接生命周期和事务边界

**建议**: 添加监控逻辑:
```java
@Override
public void close() throws java.sql.SQLException {
    long start = System.nanoTime();
    try {
        delegate.close();
        monitor.recordTransaction("CONNECTION_CLOSE", System.nanoTime() - start);
    } catch (SQLException e) {
        monitor.recordError("CONNECTION_CLOSE", System.nanoTime() - start, e);
        throw e;
    }
}
```

#### 3.2.4 【中优先级】LoggingSqlListener 日志安全隐患

**问题**: `LoggingSqlListener.java:48`
```java
log(log, "[JDBCMON] Success: {}ms - {}", elapsedMs, ctx.getSql());
```

**风险**: 
1. SQL 中可能包含敏感参数（密码、身份证号等）
2. 未对 SQL 进行脱敏处理
3. 违反编码规范（日志中禁止输出敏感信息）

**建议**: 添加 SQL 脱敏工具类:
```java
private String sanitizeSql(String sql) {
    if (sql == null) return "";
    // 脱敏 VALUES 后的内容、密码字段等
    return sql.replaceAll("(?i)VALUES\\s*\\([^)]+\\)", "VALUES(...)");
}
```

#### 3.2.5 【中优先级】SqlExecutionContext ThreadLocal 内存泄漏风险

**问题**: `SqlExecutionContext.java:19-32`
```java
private static final ThreadLocal<SqlExecutionContext> POOL =
    ThreadLocal.withInitial(SqlExecutionContext::new);

public static SqlExecutionContext acquire() {
    SqlExecutionContext ctx = POOL.get();
    ctx.reset();
    return ctx;
}

public static void release(SqlExecutionContext ctx) {
    if (ctx != null) {
        POOL.set(ctx);  // ❌ 只放回池中，未清理
    }
}
```

**风险**: 在容器环境（如 Tomcat）中，线程池复用可能导致 `SqlExecutionContext` 对象长期占用内存

**建议**: 使用 `WeakReference` 或在请求结束时明确 `remove()`:
```java
public static void release(SqlExecutionContext ctx) {
    if (ctx != null) {
        ctx.reset();
        POOL.remove();  // 明确清理
    }
}
```

#### 3.2.6 【低优先级】CompositeSqlListener 异常处理过度

**问题**: `CompositeSqlListener.java:31-37`
